# 面向大模型推理芯片的带宽—算力—容量协同设计框架

## —— 从 Workload 演进到 PPA 配比的可复算设计方法

> 本文档是对《带宽算力配比设计框架.md》的结构性修订版。修订目标不是提出一个跨模型、跨上下文、跨 batch 都固定不变的“神奇指标”，而是建立一套面向 LLM 推理芯片早期规格定义的设计框架：把模型结构、上下文长度、batch、SLA、KV cache、片上容量、利用率和 PPA 成本放进同一个分析平面，用于指导算力、带宽、容量和片上存储的协同配比。

---

# 0. 执行摘要

大模型推理芯片的核心设计问题，正在从“峰值算力够不够”转向“算力、带宽、容量、片上存储和调度利用率是否匹配”。同样的峰值算力，在短上下文 dense 模型、长上下文 GQA/MLA 模型、MoE 模型、稀疏注意力模型、SSM/Linear Attention 模型、MTP/投机解码场景下，可能面对完全不同的瓶颈。

传统 Roofline 模型可以判断某个 kernel 或 workload 在给定硬件上更偏 compute-bound 还是 memory-bound；但在 LLM 推理芯片的规格定义阶段，仅判断“当前是否受限”还不够。芯片设计需要回答更前置的问题：

1. 给定未来几类模型和服务场景，芯片每单位算力应配置多少片外带宽？
2. 需要多少片外容量和片上存储，才能支撑目标上下文长度、batch 和 SLA？
3. 随着 GQA、MLA、Sparse/Compressed Attention、SSM、MTP/Speculative Decoding 等优化技术演进，硬件设计应向哪个方向移动？
4. 现有 GPU、可重构架构、权重驻留架构或大 SRAM 架构，分别会在哪些 workload 下失衡？
5. 对于可重构推理芯片，哪些资源应该做成可重构，哪些资源应该固化，才能覆盖未来模型结构的不确定性？

本文提出一套三层设计框架：

| 层级 | 解决的问题 | 主要输出 |
|---|---|---|
| 第 1 层：Workload 需求层 | 每 token 需要多少计算、片外搬运和状态存储？ | `O_tok(C)`、`Q_tok(C,B,S)`、KV 容量需求 |
| 第 2 层：PPA 成本层 | 增加算力、带宽、片上存储、片外容量分别要付出多少面积/功耗/成本？ | `Area(P_c, BW, S, Mem)` 或多目标成本函数 |
| 第 3 层：映射利用率层 | 给定架构和调度，峰值资源有多少能变成有效吞吐？ | `u_c`、`u_b`、`u_s` 等仿真/实测利用率 |

三层合成后，框架输出的不是一个常数，而是一族可复算的设计曲线：

> 在给定模型族、上下文分布、batch 策略、SLA 和候选架构后，芯片每单位算力应配置多少带宽，即 `(BW/P_c)*` 曲线；同时给出容量、片上存储和利用率的约束边界。

在这个框架中，EBpC（Effective Bandwidth per Compute）不再被定义为一个独立的“万能指标”，而是降级为该框架中的一个派生观测量：

> **EBpC = workload 在某个设计点上的有效字节/计算需求。**  它可以用于展示趋势、比较模型和定位瓶颈，但不能单独决定芯片设计。真正用于设计决策的是三层框架合成后的设计曲线和 PPA 约束面。

# 1. 背景：为什么现在需要新的设计框架

## 1.1 LLM 推理需求正在快速变化

LLM 推理的硬件需求并不是稳定的。模型结构和推理算法正在持续改变每 token 的计算量、权重搬运量、KV cache 读写量和状态存储量：

| 技术演进 | 对 workload 的主要影响 | 对芯片设计的含义 |
|---|---|---|
| MHA → MQA/GQA | 降低 KV 存储和搬运，不显著降低 attention FLOPs | 长上下文带宽/容量压力下降，但 decode 仍可能 memory-bound |
| GQA → MLA | 大幅压缩 KV 表示，通常用额外计算换带宽 | 算力与带宽的最优配比发生移动 |
| Sliding Window / Sparse / DSA | 降低实际读取条目，未必同步降低总存储 | 索引、调度、片上缓存命中率变重要 |
| Linear Attention / SSM | 将随上下文增长的 KV 读写替换成近似常数状态 | 长上下文带宽压力显著下降，状态更新和片上存储更重要 |
| MTP / Speculative Decoding | 一次权重读取服务多个 token | 权重流被进一步摊薄，decode 的计算/带宽平衡点移动 |
| MoE | 单 token 激活参数低，但全量参数和专家调度复杂 | 容量、专家驻留、跨请求 batch 聚合成为核心问题 |

因此，面向未来 2~3 年产品周期的推理芯片，不能只为某一个固定模型或固定上下文长度设计。它需要面对一组随时间演进的 workload 分布。

## 1.2 固定“带宽—算力比”不足以指导设计

行业中常见的带宽—算力比，通常默认一组隐含条件：权重和 KV 都在片外；每 token 都流式读取权重和 KV；batch 固定；片上存储只承担 tile 内复用；attention 算法类型固定。在这些约定下，带宽—算力比可以被算成一个数。

但这些被默认冻结的参数，恰恰是芯片设计要决策的自由度：

- batch 从 1 到 128，权重流的每 token 搬运量会变化两个数量级；
- 片上存储从几十 MB 到几百 MB，会改变权重、KV、激活的驻留策略；
- MTP 或投机解码会让一次权重读取服务多个 token，击穿“每 token 搬一次权重”的默认假设；
- MLA、稀疏注意力、SSM 会改变 KV 读流量和存占用的函数形态；
- 可重构架构的真实优势取决于这些变化能否被映射到更高的有效利用率。

因此，一个通过冻结自由度才得到的常数，不能反过来指导这些自由度。设计框架需要把它们显式化，把“配比”从常数变成函数。

## 1.3 本框架与 Roofline 的关系

Roofline 的基本形式是：

```text
性能 = min(峰值算力, 算术强度 × 带宽)
```

它适合回答：“给定硬件和 workload，当前瓶颈在哪里？”

本文框架借鉴 Roofline 的 compute/memory 双约束思想，但面向的问题更靠前：

> **在芯片规格定义阶段，给定一组未来 workload 和 SLA，应该如何配置峰值算力、片外带宽、片外容量和片上存储，使得设计点在 PPA 上尽量均衡？**

因此，本文框架不是 Roofline 的替代，而是面向 LLM 推理芯片设计的扩展：

1. 将 LLM decode 的权重流、KV 流、状态流拆开建模；
2. 将 batch、上下文长度、片上容量、注意力机制作为显式变量；
3. 将 PPA 成本和映射利用率纳入同一设计平面；
4. 输出“设计曲线族”，而不是单点性能判断。

# 2. 核心洞察：配比不是常数，而是函数

## 2.1 两类流量必须拆开

Decode 阶段每 token 的片外搬运量可以一阶拆成两类：权重流和 KV/状态流。


$$
B_{tok}(C,B) = \frac{N_{act} \cdot b_W}{B} + s \cdot \Phi_{read}(C) + \varepsilon_{act}
$$


其中：

- `N_act · b_W / B` 是权重流。它会随 batch 增大而摊薄；
- `s · Φ_read(C)` 是 KV/状态流。它通常是每请求私有，不能被 batch 直接摊薄；
- `ε_act` 是激活、索引、量化/反量化、调度等附加搬运项。

这一拆分非常关键。权重流和 KV 流的复用度、容量属性、优化手段完全不同：

| 流量类型 | 是否随 batch 摊薄 | 是否随上下文增长 | 主要优化手段 | 对硬件的影响 |
|---|---:|---:|---|---|
| 权重流 | 是 | 通常不随 C 增长 | batch、MTP、投机解码、权重驻留 | 影响带宽/算力配比和片上/片外容量 |
| KV/状态流 | 通常否 | 取决于 attention 机制 | GQA、MLA、Sparse、SSM、KV 量化 | 影响长上下文带宽和容量上限 |
| 附加流量 | 视实现而定 | 视实现而定 | kernel fusion、调度、缓存、索引硬化 | 影响 `u_b` 和实际可达吞吐 |

试图对两类流量的总和寻找一个普适商，通常会失败；规律在分量里，而不是在总商里。

## 2.2 计算量也要区分含参和无参部分

每 token 计算量可一阶写为：


$$
O_{tok}(C) = 2N_{act} + O_{attn}(C)
$$


其中：

- `2N_act`：含参矩阵乘，包括 QKV/O 投影、FFN、embedding/LM head 等实际参与计算的参数；MoE 模型取单 token 激活参数；
- `O_attn(C)`：无参 attention 计算，主要包括 `qK^T` 和 `AV`，通常随上下文长度 C 增长。

以 MHA/GQA 为例：


$$
O_{attn}^{GQA}(C) \approx 4 \cdot L \cdot d \cdot C
$$


注意：GQA 降低的是 KV 存储和搬运，不等价于按同样比例降低 attention FLOPs。对很多模型而言，attention 计算仍然跟 Query 头数走。

对 MLA 吸收式计算，一阶可写为：


$$
O_{attn}^{MLA}(C) \approx 2 \cdot L \cdot H_Q \cdot C \cdot (2d_c + d_R)
$$


MLA 的本质不是“免费压缩 KV”，而是在许多实现中用额外或不同形态的计算换取 KV 字节下降。

## 2.3 EBpC 是派生量，不是完整框架

在给定模型、上下文、batch、片上容量、量化、attention 机制和调度假设后，可以定义一个 workload 层的需求比：


$$
EBpC_{wl}(C,B,S) = \frac{Q_{tok}(C,B,S)}{O_{tok}(C)}
$$


这个量可以用于：

- 比较不同模型或不同上下文长度下的字节/计算需求；
- 展示 attention 优化技术如何移动 workload 曲线；
- 判断某个 workload 更偏带宽受限还是算力受限；
- 作为设计框架中的输入量。

但它不能单独回答“芯片应该怎么设计”，因为芯片设计还需要考虑：

1. 算力、带宽、片上存储、片外容量的面积/功耗/成本单价；
2. 真实映射下算力和带宽的有效利用率；
3. TTFT、TPOT、吞吐、p99 等 SLA；
4. HBM/LPDDR 通道数、封装、NoC、SRAM bank、频率、电压等离散工程约束。

因此，EBpC 在本文中只作为派生观测量保留；真正的设计输出是三层合成后的 `(BW/P_c)*` 曲线、容量约束面和 PPA Pareto 边界。

# 3. 三层设计框架

## 3.1 第 1 层：Workload 需求层

Workload 层回答：在给定模型和部署假设下，每个 token 需要多少计算、多少片外搬运、多少状态容量。

主要输入包括：

| 符号 | 含义 | 来源 |
|---|---|---|
| `L, d, H_Q, H_KV, d_h` | 层数、隐藏维、Query/KV 头数、单头维度 | 模型配置 |
| `C` | 当前上下文长度 | 业务/运行时变量 |
| `N_act` | 单 token 实际参与矩阵乘的参数量 | 模型配置；MoE 取激活参数 |
| `N_res` | 需要片外/系统内存驻留的参数量 | dense 通常等于 N；MoE 通常接近全量参数 |
| `b_W, b_KV, b_act` | 权重、KV、激活每元素字节数 | 量化和部署选择 |
| `B` | decode batch size | 调度选择 |
| `S` | 可用于跨 token/跨层驻留的片上存储 | 芯片设计变量 |
| `Mem` | 片外容量 | 芯片/系统设计变量 |

第 1 层的输出包括：


$$
O_{tok}(C), \quad Q_{tok}(C,B,S), \quad KV_{store}(C,B), \quad Mem_{req}(C,B)
$$


它们进入后续 PPA 和 SLA 约束。

## 3.2 第 2 层：PPA 成本层

PPA 层回答：增加算力、带宽、片上存储、片外容量分别要付出多少代价。

早期设计阶段可以用一阶线性成本模型近似：


$$
Area(P_c, BW, S, Mem) = \sum_P P_c^{(P)} \alpha_P + BW \alpha_B + S \alpha_S + Mem \alpha_M
$$


其中：

| 符号 | 含义 |
|---|---|
| `P_c^(P)` | 精度 P 下的峰值算力 |
| `BW` | 片外带宽 |
| `S` | 片上存储容量 |
| `Mem` | 片外容量 |
| `α_P` | 单位算力面积/功耗/成本单价 |
| `α_B` | 单位带宽 PHY+控制器+封装成本单价 |
| `α_S` | 单位片上 SRAM 成本单价 |
| `α_M` | 单位片外容量/BOM/封装成本单价 |

该模型不是后端 signoff 模型，而是架构早期搜索模型。它用于回答：在固定面积、功耗或 BOM 预算下，算力、带宽、容量和片上存储应该如何 trade-off。

## 3.3 第 3 层：映射利用率层

映射层回答：峰值硬件资源在真实 workload、调度和编译映射下，有多少能转化为有效吞吐。


$$
u_c = \frac{\text{实际 FLOPs/s}}{P_c}, \qquad u_b = \frac{\text{实际 Bytes/s}}{BW}
$$


其中：

| 利用率 | 含义 | 典型来源 |
|---|---|---|
| `u_c` | 算力利用率，类似 MFU | 架构仿真、kernel profiling、实测 |
| `u_b` | 带宽利用率，类似 MBU | 架构仿真、访存 trace、实测 |
| `u_s` | 片上存储/NoC/调度有效性，可选扩展 | 编译器、仿真器、RTL/FPGA/硅后数据 |

`u_c`、`u_b` 不应被拍成常数。它们是 `(workload, hardware, compiler/scheduler)` 的函数。对尚不存在的候选架构，仿真器是生成这些利用率修正项的主要手段；对已有竞品，可以通过公开规格、benchmark 和实测反推近似区间。

# 4. 注意力与推理优化技术的统一参数化

## 4.1 四元组表示

不同 attention 技术不应只用“相对 MHA 的压缩率”来描述。更稳妥的方式，是用四类变量描述它对读流量、存占用、每条目字节和每条目计算的影响：


$$
B_{tok}^{KV} = s \cdot \Phi_{read}(C) + B_{idx}
$$



$$
O_{attn} = \omega \cdot \Phi_{read}(C) + O_{idx}
$$



$$
KV_{store} = s_{store} \cdot \Phi_{store}(C)
$$


| 变量 | 含义 |
|---|---|
| `Φ_read(C)` | 每 token 实际访问的 KV/状态条目数 |
| `Φ_store(C)` | 需要存储的 KV/状态条目数 |
| `s` | 每个读取条目的字节数 |
| `s_store` | 每个存储条目的字节数 |
| `ω` | 每个读取条目对应的 attention FLOPs |
| `B_idx, O_idx` | 稀疏索引、检索、调度、量化/反量化等附加开销 |

## 4.2 典型技术的影响

| 技术 | `Φ_read(C)` | `Φ_store(C)` | 字节影响 | 计算影响 | 设计含义 |
|---|---|---|---|---|---|
| MHA | `L·C` | `L·C` | 基准最高 | 基准 | 长上下文下 KV 带宽/容量快速增长 |
| MQA/GQA | `L·C` | `L·C` | KV 字节按 KV 头数下降 | attention FLOPs 近似不按同等比例下降 | 降带宽/容量，不等于降所有计算 |
| MLA | `L·C` | `L·C` | latent KV 大幅下降 | 可能增加吸收式 attention 计算 | 典型“算力换带宽” |
| SWA | `L·min(C,W)` | 视实现而定 | 读流量封顶 | attention 计算封顶 | 适合局部上下文，但全局信息需补充机制 |
| DSA/top-k | `L·k` | 通常仍需 `L·C` 或索引结构 | 实际读流量下降 | 增加索引/检索开销 | 调度和不规则访存能力重要 |
| 块压缩 | `L·C/m` | `L·C/m` | 读写与存储同步下降 | 取决于压缩 attention | 改变长上下文斜率 |
| Linear/SSM | `O(1)` | `O(1)` | 从随 C 增长变成近似常数 | 状态更新计算 | 长上下文带宽压力下降 |
| MTP/Spec Decode | 不直接改变 KV 形式 | 不直接改变 KV 形式 | 权重流按有效 token 数摊薄 | 增加验证/草稿模型计算 | 改变 decode 均衡点 |

## 4.3 组合规则

这些优化技术可以组合，但组合时要区分影响对象：

1. 量化主要乘在 `s`、`s_store`、`b_W` 上；
2. GQA/MQA 主要改变 KV 字节，不等价于同比例改变 attention FLOPs；
3. MLA 同时改变 KV 字节和 attention 计算形态；
4. 稀疏、窗口、压缩主要是对 `Φ_read(C)` 和 `Φ_store(C)` 的变换；
5. MTP/Spec Decode 主要改变权重流的摊薄方式；
6. 层间异构模型应按层求和，而不是用单一全局压缩率粗略替代。

这也是为什么本文不建议把“LLM 优化趋势”压缩成一个单独系数。更稳妥的做法，是把不同技术映射为四元组变换，再生成不同情景下的设计曲线。

# 5. 从需求到设计：均衡配比近似

## 5.1 Decode 阶段 SLA 约束

在 compute 与 memory 可充分重叠的一阶 Roofline 近似下，decode 单步时间为：


$$
t_{step} = \max\left(\frac{B \cdot O_{tok}(C)}{P_c \cdot u_c},\ \frac{B \cdot Q_{tok}(C,B,S)}{BW \cdot u_b}\right)
$$


需要满足：


$$
t_{step} \leq TPOT
$$


其中 TPOT 是每输出 token 的时延约束。

## 5.2 容量约束

对于 dense 模型，片外容量主要由权重和 KV 池决定；对于 MoE 模型，虽然单 token 激活参数量是 `N_act`，但系统通常仍需驻留或管理更接近全量的 `N_res`：


$$
N_{res} \cdot b_W + B \cdot s_{store} \Phi_{store}(C) \leq Mem
$$


这条容量约束会反过来限制 batch 上限，进而影响权重流摊薄程度。

## 5.3 均衡配比近似

在连续成本、资源可平滑缩放、compute/memory 两支路充分重叠、且其他约束不先于两支路生效的理想条件下，成本最优设计点倾向于让计算支路和带宽支路接近同时踩线：


$$
\frac{B \cdot O_{tok}(C)}{P_c \cdot u_c} \approx \frac{B \cdot Q_{tok}(C,B,S)}{BW \cdot u_b}
$$


整理得到：


$$
\left(\frac{BW}{P_c}\right)^* \approx \frac{Q_{tok}(C,B^*,S)}{O_{tok}(C)} \cdot \frac{u_c}{u_b}
$$


如果引入片上驻留优化，可将 `Q_tok` 换成 `Q_min(S,B|C)`：


$$
\left(\frac{BW}{P_c}\right)^* \approx \frac{Q_{min}(S,B^* \mid C)}{O_{tok}(C)} \cdot \frac{u_c}{u_b}
$$


这个式子是本文框架的核心接口，但它必须被理解为**均衡设计近似**，而不是严格物理定律。真实芯片会受到 HBM 通道数、封装、NoC、SRAM bank、频率、电压、布局布线、功耗墙、成本档位等离散约束影响。最终设计点应通过 PPA 模型和仿真扫描校准。

## 5.4 `B*` 不是外部常数

`B*` 不是任意给定的 batch，而是由业务请求分布、调度策略、容量约束和 SLA 共同决定的有效 batch：

```text
Mem, C, SLA, scheduler  →  B*  →  Q_tok  →  (BW/P_c)*  →  Area/PPA
```

因此，带宽算力配比不能脱离 batch 策略讨论。对于 decode，batch 决定权重流摊薄；对于 prefill，prompt 长度在某种意义上承担了类似 batch 的权重摊薄作用。

# 6. Prefill / Decode 分离与 PD 架构启示

## 6.1 两个阶段的硬件拉力不同

Prefill 和 decode 共用模型参数，但对硬件的压力不同：

| 阶段 | 主要计算形态 | 主要瓶颈倾向 | SLA 指标 | 对硬件的拉力 |
|---|---|---|---|---|
| Prefill | 矩阵 × 矩阵，大量 token 并行 | 通常更偏 compute-bound，但 MoE 短 prompt 可偏带宽 | TTFT | 拉高绝对算力 `P_c` |
| Decode | 矩阵 × 向量 / 小 batch GEMM，逐 token 自回归 | 长上下文下更偏带宽/容量受限 | TPOT、吞吐、p99 | 拉高 `BW/P_c`、容量、调度效率 |

这意味着同一颗芯片同时服务 prefill 和 decode 时，可能出现“两头不讨好”：prefill 希望高算力密度，decode 希望高带宽/容量配比和更高 batch 聚合能力。

## 6.2 PD 分离的判据

PD 分离不是架构偏好，而是由两类 SLA 锚点的张力决定：

1. TTFT 约束给出 prefill 所需的绝对算力下限；
2. TPOT / throughput 约束给出 decode 所需的带宽、容量和有效 batch；
3. 若两者在 `(P_c, BW, Mem, S)` 设计空间中的最优区域相距较远，PD 分离就有价值；
4. 若业务 prompt 较短、MoE prefill 仍带宽受限，或者 decode batch 很高，两阶段张力可能减弱。

因此，本框架应输出第二类关键图：

> **PD 分离判据图**：在 `(P_c, BW)` 或 `(BW/P_c, Mem)` 平面上同时画出 prefill 锚点和 decode 锚点，观察单芯片设计是否被两侧拉扯。

# 8. 对可重构推理芯片设计的启示

本框架对可重构芯片的启示，不是简单说明“可重构更灵活”，而是要回答：未来 LLM workload 的哪些部分会变化，哪些硬件资源需要为这些变化留出映射空间。

## 8.1 可重构价值应绑定 workload 演进

如果只是说“我们也做可重构计算”，很容易与已有可重构架构路线重叠。更有说服力的表达是：

> LLM 推理 workload 正在从固定 dense attention，演进到 GQA、MLA、Sparse/Compressed Attention、SSM、MTP/Spec Decode 混合共存。不同技术改变的是数据流、KV 访问模式、计算/访存平衡和调度粒度。可重构架构的价值，在于可以随这些变化重新映射算子、数据流和片上存储策略，从而提升有效利用率并延长芯片架构生命周期。

## 8.2 可重构不等于所有东西都可变

可重构资源应该服务于变化最大的部分，而不是让所有模块都保持完全可编程。依据本文框架，变化最明显的部分包括：

| 变化来源 | 对架构的影响 | 可重构设计关注点 |
|---|---|---|
| Attention 机制变化 | KV 读取条目、字节、计算形态变化 | 数据流、访存模式、算子组合可重构 |
| 稀疏/压缩/索引机制 | 不规则访问和索引开销上升 | 调度器、DMA、片上缓存、索引单元协同 |
| MTP/Spec Decode | 权重流摊薄方式变化 | 多 token 并行、验证路径、pipeline 调度 |
| MoE | 专家选择、容量驻留、跨请求聚合复杂 | 专家调度、片上/片外搬运编排、batch 聚合 |
| 量化格式演进 | FP8/FP4/INT4/NF4 等混合精度 | 数值格式转换、混合精度算术核、累加路径 |

因此，可重构芯片的差异化不应只讲“阵列灵活”，而应讲：

1. 用 workload 需求曲线决定算力、带宽、容量和片上存储配比；
2. 用可重构数据流覆盖 attention/SSM/MoE/MTP 的结构变化；
3. 用仿真器生成 `u_c/u_b` 修正项，避免只看峰值规格；
4. 用 PPA 模型选择哪些能力硬化、哪些能力保留可重构。

## 8.3 建议对外叙事

建议在 BP 中采用克制表述：

> 我们的核心差异不是单点算子或单个硬件模块，而是建立了 workload 演进、架构配比、PPA 成本和映射利用率之间的闭环。可重构架构是实现该闭环的硬件载体，使芯片能够适应未来 LLM 模型结构和推理算法的持续变化。

这个叙事比“我们提出了一个新指标”更稳，也更容易与团队既有大算力芯片流片经验、仿真工具和可重构技术路线形成闭环。

# 10. 适用范围与边界

本文框架定位于**芯片架构早期设计和 BP 技术方法论表达**，不替代完整仿真、RTL 实现、后端 PPA signoff 或硅后实测。

需要明确以下边界：

1. **一阶重叠假设**：正文采用 `max(compute_time, memory_time)` 近似，真实执行可能介于 max 与 sum 之间；
2. **利用率是函数而不是常数**：`u_c`、`u_b` 随模型、batch、context、kernel、调度和硬件结构变化；
3. **成本模型是早期近似**：线性面积/功耗模型只用于设计空间搜索，真实 PPA 存在台阶和非线性；
4. **`Q_min` 是驻留分配近似**：真实系统还存在预取、流水、压缩、缓存替换和跨层调度；
5. **KV 命中率不一定均匀**：稀疏检索、热点上下文、检索增强和多轮对话会导致分布偏斜；
6. **多芯片系统需重新定义边界**：片间链路、HBM 分布、专家并行、张量并行、流水并行会改变 `BW` 与 `S` 的归属；
7. **模型技术路线不确定**：MLA、Sparse、SSM、MTP 等应作为情景变量，而非单一确定预测；
8. **BP 表达与研发模型应分层**：BP 正文展示洞察、曲线和落点；技术附录保留公式、假设和复算过程。

# 附录 A：公式与推导细节

## A.1 Decode 计算量


$$
O_{tok}(C) = 2N_{act} + O_{attn}(C)
$$


MHA/GQA：


$$
O_{attn}^{GQA}(C) \approx 4LdC
$$


MLA 吸收式：


$$
O_{attn}^{MLA}(C) \approx 2LH_QC(2d_c + d_R)
$$


## A.2 Decode 搬运量


$$
B_{tok}(C,B) = \frac{N_{act}b_W}{B} + s\Phi_{read}(C) + \varepsilon_{act}
$$


KV 存占用：


$$
KV_{store}(C,B) = B \cdot s_{store} \Phi_{store}(C)
$$


容量约束：


$$
N_{res}b_W + B \cdot s_{store}\Phi_{store}(C) \leq Mem
$$


## A.3 注意力四元组


$$
B_{tok}^{KV} = s\Phi_{read}(C) + B_{idx}
$$



$$
O_{attn} = \omega\Phi_{read}(C) + O_{idx}
$$



$$
KV_{store} = s_{store}\Phi_{store}(C)
$$


典型取值：

| 技术 | `Φ_read(C)` | `Φ_store(C)` | `s` | `ω` |
|---|---|---|---|---|
| MHA | `L·C` | `L·C` | `2H_Qd_hb` | `4H_Qd_h` |
| MQA/GQA | `L·C` | `L·C` | `2H_KVd_hb` | `4H_Qd_h` |
| MLA | `L·C` | `L·C` | `(d_c+d_R)b` | `2H_Q(2d_c+d_R)` |
| SWA | `L·min(C,W)` | `L·min(C,W)` 或宿主策略 | 继承宿主 | 继承宿主 |
| DSA/top-k | `L·k` | 通常 `L·C` + 索引 | 继承宿主 | 继承宿主 + 索引开销 |
| 块压缩 | `L·C/m` | `L·C/m` | 压缩条目字节 | 对应压缩 attention |
| Linear/SSM | `O(1)` | `O(1)` | 状态字节 | 状态更新 FLOPs |

## A.4 片上驻留下的 `Q_min`

引入片上可用于跨 token/跨层驻留的存储 `S` 后，每 token 最小片外流量可近似写成：


$$
Q_{min}(S,B \mid C) = \min_{S_W + S_{KV} \leq S}
\left[
\frac{\max(N_{act}b_W - S_W,0)}{B}
+ s\Phi_{read}(C)\left(1 - \frac{S_{KV}}{B s\Phi_{store}(C)}\right)_+
\right]
$$


解释：

- `S_W` 用于权重驻留，减少权重流；
- `S_KV` 用于 KV/状态驻留，减少 KV 片外读取；
- 权重驻留收益随 batch 被摊薄，KV 驻留收益取决于每请求私有状态和命中率；
- 真实系统还应考虑预取、流水、替换策略和跨层调度。

## A.5 Decode 均衡公式

Decode 单步时间：


$$
t_{step} = \max\left(\frac{B O_{tok}}{P_cu_c}, \frac{B Q_{min}}{BWu_b}\right)
$$


均衡近似：


$$
\frac{B O_{tok}}{P_cu_c} \approx \frac{B Q_{min}}{BWu_b}
$$


得到：


$$
\left(\frac{BW}{P_c}\right)^* \approx \frac{Q_{min}(S,B^* \mid C)}{O_{tok}(C)} \cdot \frac{u_c}{u_b}
$$


## A.6 Prefill 计算量

Prompt 长度为 `S_p` 时，prefill 总计算量一阶为：


$$
O_{pre}^{lin} = 2N_{act}S_p
$$



$$
O_{pre}^{attn} \approx 2LdS_p^2
$$



$$
O_{pre}(S_p) = 2N_{act}S_p + 2LdS_p^2
$$


MLA 吸收式可替换 attention 项：


$$
O_{pre}^{attn,MLA} \approx LH_QS_p^2(2d_c+d_R)
$$


## A.7 Prefill 搬运量


$$
Q_{pre}(S,B_p|S_p) =
N_{res}b_W
+ s\Phi_{store}(S_p)B_p
+ Ls\frac{S_p}{2}n_{pass}(S)B_p
+ \kappa LS_pd b_{act}B_p
$$


其中：

- 第一项：权重在整个 prefill 阶段读取一次；MoE 场景应谨慎区分 `N_act` 与 `N_res`；
- 第二项：写出 prompt 对应的 KV cache；
- 第三项：FlashAttention 类分块下的 KV 重读；
- 第四项：激活溢出读写，`κ` 可取 0 到 2 的区间估计。

## A.8 Prefill compute-bound 判据

当权重项主导搬运、含参项主导计算时：


$$
AI_{pre} \approx \frac{2N_{act}S_p}{N_{res}b_W}
$$


与硬件 ridge point `R` 比较，得到 compute-bound 的 prompt 长度阈值：


$$
S_p^* = \frac{R b_W}{2}\cdot\frac{N_{res}}{N_{act}}
$$


Dense 模型中 `N_res ≈ N_act`，阈值通常较低；MoE 模型中 `N_res/N_act` 可能显著大于 1，短 prompt prefill 未必 compute-bound。

## A.9 完整优化问题


$$
\min_{P_c,BW,S,Mem,B}\ Cost(P_c,BW,S,Mem)
$$


约束：


$$
t_{step} \leq TPOT
$$



$$
t_{pre} \leq TTFT
$$



$$
N_{res}b_W + B s_{store}\Phi_{store}(C) \leq Mem
$$


其中成本函数可以是面积、功耗、BOM，或多目标加权函数：


$$
Cost = \lambda_A Area + \lambda_P Power + \lambda_B BOM
$$
